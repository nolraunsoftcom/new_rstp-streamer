## 0. 소스/도구 구성 (하이브리드)

| 소스 | 용도 | 비고 |
|---|---|---|
| 실장비 `rtsp://169.254.4.1:8900/live` | 충실도(지연/HW디코드/실재연결), 세션용량 측정 | READ-ONLY |
| 로컬 합성 RTSP `rtsp://127.0.0.1:8554/cam<N>` | 20ch 스케일·펜아웃·렌더·녹화 동시성(재현가능) | mediamtx + ffmpeg 루프 발행 |

- 합성 서버: `mediamtx`(미설치 시 `brew install mediamtx`). 설정은 `~/.ziilab/mediamtx.yml` 재사용/최소화.
- 발행: `ffmpeg -re -stream_loop -1 -i tests/fixtures/h264.mkv -c copy -f rtsp rtsp://127.0.0.1:8554/cam<N>` (채널 수만큼, 또는 1개 발행 후 동일 path를 N채널이 구독 = mediamtx 펜아웃).
- 클라이언트 차단: `pfctl` 앵커로 장비 IP/포트를 일시 차단·해제(**sudo 필요** — 세션에서 `!sudo ...`로 실행).

**스크립트 (ops/):**
- `device-battery.sh` — 기존: 1h 스트리밍+녹화+캡처, 토글 스트레스 (M3 핵심)
- `diskfull-test.sh` — 기존: ENOSPC 재현 D10 백오프 검증 (✅ 통과)
- `synthetic-source.sh` — 신규: mediamtx 기동 + N개 ffmpeg 루프 발행 (start/stop)
- `reconnect-test.sh` — 신규: pfctl 차단/해제 사이클 + 재연결 로그 판정
- `scale-test.sh` — 신규: 20ch(하이브리드) 구동, CPU/RSS/채널독립성/올다운경보 판정
- `session-capacity.sh` — 신규: 실장비 동시 RTSP 세션 상한 측정

## Phase 1 — M1 단일 채널 재생 코어 (≈1h15m)

**대상 기능:** RTSP-TCP 연결, FFmpeg demux, VideoToolbox HW디코드(+SW폴백), zero-copy NV12 렌더, 상태머신(Idle→…→Streaming/Stalled/Reconnecting/Failed), 패킷흐름 stall 감지, 저지연.

- [ ] **1.1 연결/재생(수동·시각):** 앱 실행 → 1채널 연결 → 영상 표시 확인. **PASS:** 프레임 표시, 깨짐 없음.
- [ ] **1.2 디코드 경로 로그:** stderr `decode path = HW` + `render path = NV12 zero-copy`. **PASS:** HW+zero-copy 경로.
- [ ] **1.3 저지연 체감(수동):** libVLC 대비 거의 실시간. **PASS:** 1초 이상 누적지연 없음.
- [ ] **1.4 패킷흐름 표시(수동):** 끊기면 UI 표시기가 즉시 반응. **PASS:** stall 시 즉시 가시화.
- [ ] **1.5 재연결 사이클(자동, pfctl):** `reconnect-test.sh` — 장비 5회 차단(각 15s)/해제. **PASS:** 매 차단 후 Stalled→Reconnecting→재연결, 매 복구 후 Streaming, reconnectAttempts 누적 없음(framePresented/패킷 리셋), 영구 Failed 없음.
- [ ] **1.6 저빈도 재시도(D1, 자동):** 장비 60s+ 차단 → maxAttempts 초과 시 Failed(저빈도 60s) → 해제 후 자동 복귀. **PASS:** Failed가 종료상태 아님(60s 후 OpenSource), 복구 시 Streaming.
- [ ] **1.7 1시간 단일채널 안정성(자동):** `device-battery.sh` Phase1 단일채널 부분 또는 전용 1h 런. **PASS:** RSS 평탄(누수 0), 크래시 0, 자연 stall 발생해도 자동 복구.

## Phase 2 — M2 멀티채널 그리드 + UI parity (≈45m)

**대상 기능:** 최대 20ch(soft 32), 앱내 펜아웃, 그리드 레이아웃(1/2/4/9/16…), UI parity(영상관리시스템·아이콘·회색빈셀·3패널), 단일 RepaintClock, 채널독립성, 올다운경보, 오프스크린 폴링스킵(D9).

- [ ] **2.1 실장비 세션상한 측정(자동):** `session-capacity.sh` — 동시 RTSP 세션을 1→N 증가시키며 안정 상한 확인. **출력:** 장비 안정 세션 수 K.
- [ ] **2.2 하이브리드 20ch 구성:** 장비 K채널 + 합성 (20−K)채널로 channels.json 구성.
- [ ] **2.3 그리드 레이아웃(수동·시각):** 1/2/4/9/16/20 분할 클릭 → 하단까지 꽉 채움, 빈 셀 회색. **PASS:** 레거시와 동일 분할·색·문구.
- [ ] **2.4 UI parity(수동·시각):** 앱이름 "영상관리시스템", 로고/아이콘, 3패널. **PASS:** 레거시와 동일.
- [ ] **2.5 채널 독립성(자동):** 한 채널 차단(pfctl)이 다른 채널 영상·녹화에 무영향. **PASS:** 타채널 Streaming 유지.
- [ ] **2.6 올다운 경보(자동/수동):** 전 채널 차단 → 전체 다운 경보 1회. **PASS:** 경보 발생, 복구 시 해제.
- [ ] **2.7 성능(자동):** 20ch 구동 중 CPU% / RSS 샘플. **PASS:** CPU 100%대 이하(zero-copy 경로), 폴링/타이머 폭주 없음, 오프스크린 페이지는 폴링 스킵 확인.

## Phase 3 — M3 녹화/스냅샷 (≈1h30m, 대부분 자동)

**대상 기능:** 채널별 동시 녹화(MKV remux, 화면=녹화 동일), 디코드원본 PNG 스냅샷, 세그먼트 롤오버(10분), 재연결 생존(드롭=세그먼트종료/복구=새세그먼트), MKV 크래시내성(D7), 녹화-재생 격리(D8), 디스크오류 가시화+백오프(D10), 좀비녹화 차단.

- [ ] **3.1 녹화 충실도(자동):** `device-battery.sh` Phase1 — 1h 연속녹화, ffprobe 재생가능 + 화면=녹화 패킷수≈디코드프레임수. **PASS:** 전 세그먼트 PLAYABLE, 손상 0.
- [ ] **3.2 스냅샷(자동):** `--snapshot-every=20` PNG 다수. **PASS:** PNG 손상 0, 디코드원본 해상도.
- [ ] **3.3 세그먼트 롤오버:** 10분 경계 stop+새경로 start, 경로충돌 없음(ms+seq). **PASS:** 경계마다 새 파일, truncate 0.
- [ ] **3.4 녹화 재연결 생존(자동, pfctl):** 녹화 중 차단/해제 반복 → 드롭=현세그먼트종료(armed유지), 복구=새세그먼트. **PASS:** 끊김마다 녹화 지속(좀비/영구정지 없음), 각 세그먼트 재생가능.
- [ ] **3.5 토글 스트레스(자동):** `device-battery.sh` Phase2 — 8s 토글 반복. **PASS:** 크래시 0, 좀비경고 0, 세그먼트 무손상.
- [ ] **3.6 디스크오류 백오프(자동, ✅완료):** `diskfull-test.sh` — ENOSPC. **PASS(기검증):** .mkv 유계(5), disarm 1, churn 정지, 크래시 0.
- [ ] **3.7 MKV 크래시내성(D7, 자동):** 녹화 중 앱 SIGKILL → trailer 없어도 ffprobe 부분복구. **PASS:** 패킷 복구 가능(REC-IT 169·170과 일치).

## Phase 4 — 통합 스트레스 (≈1h+, 자동)

- [ ] **4.1 20ch 종합 스트레스(자동):** 하이브리드 20ch + 전채널 녹화 + 20s 스냅샷 + 재연결 사이클(pfctl 주기 차단) 1h+. **PASS:** RSS 평탄(누수 0), 크래시 0, 전 채널 녹화 세그먼트 무손상, 재연결 후 전부 복구, 올다운/개별다운 경보 정상.

## 판정/기록

- 각 Phase 결과는 `logs/m1m3-<phase>-<ts>/summary.txt`에 적재, 종합은 `docs/m1-m3-acceptance.md`.
- **실패 시:** 결함 기록 → 리펙토링(subagent+TDD) → 재테스트 → 통과 후 진행.
- 자동 스크립트는 비차단 백그라운드로 돌리고 완료 시 요약을 분석한다.

## 알려진 한계(정직)

- **Fix2(오프스크린+stall 누적→Failed 비대칭):** 장비 READ-ONLY라 30회 stall 누적을 실장비로 시간내 재현 불가 → 결정적 단위테스트 3건이 1차 증명. Phase 1.5/4.1은 패킷기반 리셋 경로의 무회귀를 확인.
- **합성 소스(`-c copy` 루프):** 코덱/타임스탬프가 장비와 다를 수 있어 충실도(지연/HW경로)는 실장비 기준으로만 판정. 스케일/펜아웃/녹화동시성은 합성으로 판정.
